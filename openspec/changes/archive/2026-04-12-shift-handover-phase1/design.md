## 上下文

**当前状态**：Greenfield 项目，仅有需求文档和 OpenSpec 配置，无实现代码。

**业务背景**：北京大学国际医院需要医护智能交接班管理系统，第一阶段聚焦核心交接班流程和 AI 报告生成，暂不对接外部系统（使用模拟数据）。

**技术约束**：
- 前端：Vue 3 + TypeScript + Vite
- 后端：Spring Boot 3 + JDK 17 + MyBatis-Plus
- 数据库：MySQL 8.0，缓存：Redis 7.x
- AI 服务：需支持国内云服务或私有化部署

**关键干系人**：
- 临床医生（最终用户）- 关注易用性和报告质量
- 信息科（运维）- 关注系统稳定性和可维护性
- 医务处（管理）- 关注交接班规范性和可追溯性

## 目标 / 非目标

**目标：**
1. 实现完整的交接班工作流（交班发起→报告生成→接班确认→归档）
2. AI 自动生成符合 SBAR 规范的交班报告
3. 支持患者数据手动录入和模拟
4. 自动计算 MEWS、Braden、跌倒风险评分
5. 待办事项提取和站内推送

**非目标：**
- 与 HIS/LIS/PACS/CDR 等真实系统对接（Phase 2）
- 语音交互功能（Phase 2）
- 质控引擎和数据分析看板（Phase 2）
- 移动端功能（Phase 3）

### 2. AI 报告生成：混合模式（预生成 + 按需刷新）

```
┌─────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────┤
│                                                         │
│  凌晨 6:00 定时任务                                      │
│  └─► 为所有在院患者生成交班报告草稿                      │
│       └─► 缓存至 Redis (TTL 2 小时)                      │
```

**理由**：
- 预生成减少医生等待时间

### 3. 数据模型设计：科室维度交班

**核心设计原则**：一次交班 = 整个科室所有患者，减少医生操作次数。

```
┌─────────────────────────────────────────────────────────────────┐
│                    数据库 ER 图 - 科室维度交班                   │
└─────────────────────────────────────────────────────────────────┘

┌──────────────────┐         ┌──────────────────────┐
│ shift_handover   │         │ handover_patient     │
│ (科室交班主表)    │         │ (交班患者明细)        │
├──────────────────┤         ├──────────────────────┤
│ id               │◄────────│ handover_id          │
│ department_id    │         │ patient_id           │
│ department_name  │         │ bed_number           │
│ from_doctor_id   │──┐      │ patient_name         │
│ to_doctor_id     │  │      │ is_critical          │
│ handover_date    │  │      │ critical_reason      │
│ shift            │  │      │ situation            │
│ status           │  │      │ background           │
│ summary_json     │  │      │ assessment           │
│ created_at       │  │      │ recommendation       │
└──────────────────┘  │      │ mews_score           │
        │             │      │ braden_score         │
        │             │      │ fall_risk            │
        ▼             │      └──────────────────────┘
┌──────────────────┐  │              │
│ handover_todo    │  │              │
│ (待办事项)        │  │              │
├──────────────────┤  │              │
│ id               │  │              │
│ handover_id      │──┘              │
│ patient_id       │◄────────────────┘
│ patient_bed      │
│ content          │
│ due_time         │
│ priority         │
│ status           │
└──────────────────┘

┌──────────────────┐         ┌──────────────────────┐         ┌──────────────────┐
│ department       │         │ doctor_department    │         │ user             │
├──────────────────┤         │ (医生-科室关联表)     │         ├──────────────────┤
│ id               │◄────────│ department_id        │────────►│ id               │
│ name             │         │ doctor_id            │         │ name             │
│ code             │         │ is_primary           │         │ role             │
│ bed_count        │         └──────────────────────┘         └──────────────────┘
└──────────────────┘

┌──────────────────┐
│ patient          │
├──────────────────┤
│ id               │
│ department_id    │
│ bed_number       │
│ name             │
│ diagnosis        │
│ ...              │
└──────────────────┘

注：user 表不再直接存储 department_id，改为通过 doctor_department 关联表实现多对多关系。
```

**关键设计点**：

```
┌─────────────────────────────────────────────────────────────────┐
│              科室交班 vs 患者交班对比                            │
└─────────────────────────────────────────────────────────────────┘

  ❌ 患者维度交班（错误）：
  ├── 科室 30 名患者 = 发起交班 30 次
  ├── 接班确认 30 次
  └── 医生工作量大，效率低

  ✅ 科室维度交班（正确）：
  ├── 科室 30 名患者 = 发起交班 1 次
  ├── 一次聚合所有患者信息
  ├── 重点患者详细报告，普通患者简要报告
  └── 接班确认 1 次，高效便捷

  数据结构：
  shift_handover（主表）
  ├── 一条记录 = 一次科室交班
  └── summary_json 存储科室统计信息
  
  handover_patient（明细表）
  ├── 一条记录 = 一个患者的交班信息
  └── 关联到 shift_handover
  
  handover_todo（待办表）
  ├── 一条记录 = 一个待办事项
  └── 关联到具体患者和科室交班
```

**理由**：
- 以科室为维度，减少医生操作次数
- 支持分层报告（重点患者详细，普通患者简要）
- 待办事项关联到具体患者，便于追踪
- 一次确认完成整个科室交接

## 风险 / 权衡

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI 生成内容不准确 | 医疗安全隐患 | 强制人工确认环节，AI 生成内容明确标注"草稿" |
| 外部 API 不稳定 | 报告生成失败 | 本地缓存 lastKnownGood，降级为模板填充 |
| 模拟数据与真实场景差异 | Phase 2 对接困难 | 数据模型设计预留扩展字段，接口层抽象 |
| 医生抵触 AI 生成 | 使用率低 | 简化编辑流程，支持一键采纳/修改 |
| 交班高峰期性能瓶颈 | 用户体验差 | Redis 缓存 + 预生成策略，限流保护 |

```
┌─────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────┘

  用户输入账号密码
```

- 后端：Spring Security + JWT + Kaptcha（图形验证码）
- 前端：验证码图片自动刷新、Token 本地存储（localStorage）

### 10. 多科室医生支持：科室上下文切换

**背景**：医生可能在多个科室工作（如周一三五在心内科，周二四在神经内科），需要在前端切换"当前科室"上下文。

**核心原则**：
- 医生只为自己工作的科室交班，不是每个科室都交一次
- 切换科室后，所有数据按当前科室过滤
- 登录后默认进入主科室（is_primary = true）

#### 数据模型变更

```
┌─────────────────────────────────────────────────────────────────┐
│                    医生-科室多对多关系                           │
└─────────────────────────────────────────────────────────────────┘

【原设计】用户-科室 1:N（一个医生只属于一个科室）：

   ┌──────────────┐         ┌──────────────┐
   │    user      │  N:1    │  department  │
   ├──────────────┤ ──────▶ ├──────────────┤
   │ department_id│         │              │
   └──────────────┘         └──────────────┘
   

【新设计】用户-科室 M:N（一个医生可在多个科室工作）：

   ┌──────────────┐         ┌──────────────────┐         ┌──────────────┐
   │    user      │         │ doctor_department│         │  department  │
   ├──────────────┤         ├──────────────────┤         ├──────────────┤
   │ id           │◄────────│ doctor_id        │────────▶│ id           │
   │ name         │         │ department_id    │         │ name         │
   │ role         │         │ is_primary       │         │ code         │
   └──────────────┘         │ schedule         │         │ bed_count    │
                            └──────────────────┘         └──────────────┘
                            
   示例数据：
   ┌───────────┬────────────────┬────────────┐
   │ doctor_id │ department_id  │ is_primary │
```

#### 前端 Store 扩展

```typescript
// authStore 扩展
interface UserInfo {
  id: number
  name: string
  role: string
  departments: Department[]  // 用户所属的所有科室
}

interface Department {
  id: number
  name: string
  isPrimary: boolean
}

// 新增状态
state: {
  currentDepartmentId: number | null  // 当前选中的科室
}

// 新增 getter
getters: {
  currentDepartment: (state) => 
    state.userInfo?.departments.find(d => d.id === state.currentDepartmentId),
  canSwitchDepartment: (state) => 
    (state.userInfo?.departments.length ?? 0) > 1
}

// 新增 action
actions: {
  switchDepartment(departmentId: number) {
    this.currentDepartmentId = departmentId
    localStorage.setItem('currentDepartmentId', String(departmentId))
    // 触发数据刷新
  }
}
```

#### 默认科室选择策略

1. **首次登录**：选择主科室（is_primary = true）
2. **再次登录**：恢复上次选择的科室（localStorage 缓存）
3. **缓存失效**：回退到主科室

**技术实现**：
- 后端：新增 `doctor_department` 关联表
- 后端 API：`GET /api/user/departments` 返回医生科室列表
- 前端：TopNav 添加科室切换下拉框
- 前端：各页面按 `currentDepartmentId` 过滤数据

**理由**：
- 支持医生跨科室执业的真实场景
- 减少误操作（不会看到其他科室患者）
- 符合医院排班实际情况
