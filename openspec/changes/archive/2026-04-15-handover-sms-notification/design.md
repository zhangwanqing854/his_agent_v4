## 上下文

当前交班提流程：用户在系统发起交班 → 选择接班医生 → 点击"提交交班" → 状态从 DRAFT 变为 PENDING → 接班医生需主动登录系统查看待处理交班。

问题：接班医生可能不会及时查看系统，导致交班处理延迟。

相关数据：
- `his_staff.phone` - 接班医生电话号码（已存在）
- `shift_handover.to_doctor_id` - 接班医生 ID（已存在）
- `shift_handover.status` - 交班状态

## 目标 / 非目标

**目标：**
- 设计可扩展的短信服务接口，支持后续接入不同短信平台
- 在交班提交时自动发送短信通知接班医生
- 短信发送失败不影响交班提交流程（异步发送）
- 记录短信发送日志便于追溯
- 提供前端配置管理界面，管理员可动态配置短信平台参数

**非目标：**
- 不实现短信模板管理功能（使用固定模板）
- 不实现短信发送失败自动重试机制（第一版本）
- 不实现短信发送历史查看界面（第一版本）
- 不支持批量短信发送

## 决策

### 1. 短信服务接口设计

**选择：使用策略模式 + 接口抽象**

```java
public interface SmsService {
    SmsResult send(String phone, String content);
}
```

**理由：**
- 支持后续无缝切换不同短信平台（阿里云/腾讯云/医院自有平台）
- 只需实现新接口，无需修改业务代码
- 配置文件决定使用哪个实现

**替代方案：**
- 直接调用阿里云 SDK → 扩展性差，换平台需改动业务代码
- 使用 Spring Event → 增加复杂度，第一版本保持简单

### 2. 短信发送时机

**选择：异步发送（不阻塞交班提交）**

**理由：**
- 短信 API 可能超时，不应阻塞主流程
- 即使短信失败，交班仍应成功提交
- 系统稳定性优先

**实现方式：**
- 方案 A：使用 `@Async` 注解 + Spring 异步任务 → 选择此方案（简单）
- 方案 B：使用消息队列（RabbitMQ/Kafka） → 过度设计，第一版本不采用

### 3. 短信内容格式

**选择：固定模板格式**

```
【北京大学国际医院】您有新的交接班任务待处理。科室：{科室名}，交班医生：{医生名}，患者数：{数量}，请及时登录系统处理。
```

**理由：**
- 符合短信签名规范
- 包含关键信息便于快速了解
- 无需模板管理功能

### 4. 短信日志存储

**选择：新增 `sms_log` 表记录发送日志**

**表结构：**
```sql
CREATE TABLE sms_log (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    handover_id BIGINT NOT NULL,
    phone VARCHAR(20) NOT NULL,
    content TEXT NOT NULL,
    status VARCHAR(20) NOT NULL, -- SUCCESS/FAILED
    error_message TEXT,
    sent_at DATETIME,
    created_at DATETIME NOT NULL
);
```

**理由：**
- 便于追溯短信发送情况
- 方便后续排查问题
- 不影响现有表结构

### 5. 短信配置存储

**选择：数据库存储 + 前端管理界面**

**表结构：**
```sql
CREATE TABLE sms_config (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    config_key VARCHAR(50) NOT NULL UNIQUE,
    config_value TEXT NOT NULL,
    description VARCHAR(200),
    is_sensitive BOOLEAN DEFAULT FALSE,  -- 敏感字段（如密钥），前端不显示完整值
    updated_at DATETIME NOT NULL,
    updated_by BIGINT,  -- 更新人 ID
    created_at DATETIME NOT NULL
);
```

**预设配置项：**
| config_key | 说明 | is_sensitive |
|------------|------|--------------|
| enabled | 是否启用短信功能 | false |
| provider | 短信平台 (aliyun/tencent) | false |
| aliyun_access_key_id | 阿里云 AccessKey ID | true |
| aliyun_access_key_secret | 阿里云 AccessKey Secret | true |
| aliyun_sign_name | 短信签名 | false |
| aliyun_template_code | 短信模板 code | false |

**理由：**
- 管理员可在前端界面动态修改配置，无需重启服务
- 敏感字段标记，前端显示时脱敏处理
- 支持配置变更历史追溯

**替代方案：**
- application.yml 配置 → 修改需重启服务，不灵活
- 环境变量 → 同样需重启，且不便于管理

### 6. 前端配置管理界面

**选择：独立配置页面 + 系统设置入口**

**页面设计：**

```
┌─────────────────────────────────────────────────────────────┐
│  系统设置 > 短信配置                                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  □ 启用短信通知功能                                           │
│                                                              │
│  短信平台：○ 阿里云   ○ 腾讯云                                │
│                                                              │
│  ──── 阿里云配置 ────                                         │
│  AccessKey ID:    [ak****************xx]  [显示/隐藏]         │
│  AccessKey Secret: [****************]    [显示/隐藏]         │
│  短信签名:         [北京大学国际医院]                         │
│  模板 Code:        [SMS_123456789]                           │
│                                                              │
│  [保存配置]  [测试发送]                                       │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│  发送状态统计                                                 │
│  今日发送：12 条  成功：11  失败：1                           │
└─────────────────────────────────────────────────────────────┘
```

**功能点：**
- 启用/禁用开关
- 平台选择（阿里云/腾讯云）
- 配置参数输入（敏感字段脱敏显示）
- 保存配置按钮
- 测试发送按钮（验证配置是否正确）
- 简单统计展示（可选）

**API 接口：**
| 接口 | 说明 |
|------|------|
| GET /api/sms-config | 获取当前配置 |
| PUT /api/sms-config | 更新配置 |
| POST /api/sms-config/test | 测试发送短信 |

**理由：**
- 直观易用，管理员无需技术背景
- 敏感字段脱敏保护安全
- 测试发送验证配置有效性

## 风险 / 权衡

### 风险：短信平台费用
→ **缓解：** 短信费用由医院承担，需与医院确认已开通短信服务

### 风险：接班医生电话号码为空
→ **缓解：** 发送前校验 phone 字段，为空则跳过发送，记录日志

### 风险：短信发送失败
→ **缓解：** 异步发送 + 日志记录，失败不影响交班流程；后续可查看日志手动通知

### 风险：短信内容合规
→ **缓解：** 使用医院官方签名，内容不包含敏感医疗信息，仅提醒登录系统处理

### 风险：敏感配置泄露
→ **缓解：** 前端显示敏感字段时脱敏（仅显示前4位），完整值需点击"显示"按钮

## 迁移计划

### 部署步骤

1. 数据库执行 SQL 创建 `sms_log` 和 `sms_config` 表
2. 初始化 `sms_config` 表默认配置数据
3. 部署后端代码（SmsConfig API）
4. 部署前端代码（短信配置页面）
5. 在路由中添加配置页面入口（系统设置）
6. 管理员登录前端配置短信平台参数
7. 测试发送功能验证配置正确
8. 生产环境启用

### 回滚策略

- 在前端配置页面禁用短信功能即可关闭
- 删除 `sms_log` 和 `sms_config` 表（可选）
- 移除短信相关代码（如需完全回滚）
- 移除前端配置页面及路由

## 开放问题

1. **短信平台选择**：医院是否已有短信平台？若无，推荐阿里云短信服务（稳定、API完善）
2. **短信模板审批**：正式短信模板需向运营商报备审批，需确认医院是否已备案
3. **发送频率限制**：是否需要对同一医生设置短信发送间隔（避免频繁通知）
4. **配置权限控制**：是否限制只有特定角色（如系统管理员）才能修改短信配置
