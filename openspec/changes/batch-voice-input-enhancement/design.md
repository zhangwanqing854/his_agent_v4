## 上下文

当前语音录入功能（VoiceInputDialog.vue）仅支持单人模式，医护人员交班时需逐个患者点击语音录入按钮。实际交班场景中，医护人员往往需要连续录入5-10个患者病情，现有流程效率较低。

**约束条件：**
- 语音识别使用Web Speech API（浏览器原生），无需额外后端服务
- 床号/姓名匹配依赖已加载的患者列表数据
- 医护人员口述习惯：通常按床号顺序，格式如"2床张三，病情稳定，今日出院准备"

## 目标 / 非目标

**目标：**
- 支持一次语音录入多个患者病情信息
- 自动解析床号/姓名，匹配对应患者记录
- 提供匹配结果预览，支持手动修正
- 保持现有单人语音录入功能不变

**非目标：**
- 不优化语音识别算法（使用浏览器原生API）
- 不新增语音指令控制功能
- 不修改后端语音处理逻辑

## 决策

### 1. 批量解析策略

**选择：正则表达式解析 + 模糊匹配**

解析语音转写文本中的床号/姓名模式，使用正则表达式提取患者标识，然后与当前患者列表进行匹配。

**备选方案：**
- AI语义解析：准确度高，但增加后端依赖和响应延迟
- 关键词分割：简单分割，但无法处理复杂表述

**理由：** 正则解析成本低、响应快，医护口述格式相对固定（床号+姓名），准确度可接受。

### 2. 交互流程设计

**选择：录音 → 实时转写 → 停止 → 解析匹配 → 预览确认 → 批量填充**

用户录音时可实时查看转写内容，停止后自动解析匹配，显示预览界面供确认/修正，确认后批量填充到各患者记录。

**备选方案：**
- 录音中即时匹配：实时解析匹配，但可能导致频繁更新影响体验
- 录音完成后一次性处理：无实时反馈，用户无法确认内容准确性

**理由：** 实时转写让用户确认内容，预览界面解决匹配准确性问题。

### 3. 匹配失败处理

**选择：预览界面标记未匹配项，用户手动选择患者**

解析后未找到匹配患者的条目，在预览界面高亮显示，用户可下拉选择对应患者。

### 4. 组件结构

**选择：扩展VoiceInputDialog，新增BatchMatchPreview子组件**

在现有组件内增加模式切换（单人/批量），批量模式下显示匹配预览界面。

**备选方案：**
- 创建独立的BatchVoiceInputDialog组件：代码复用困难
- 完全重构现有组件：改动大，风险高

**理由：** 扩展模式复用现有录音逻辑，降低改动范围。

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 床号/姓名表述不规范导致匹配失败 | 提供示例引导 + 预览界面手动修正 |
| 同名患者导致匹配混淆 | 预览界面显示床号+姓名+诊断，便于区分 |
| 语音转写实时更新影响性能 | 使用debounce限制更新频率 |
| 用户习惯单人模式 | 默认保持单人模式，批量模式可选开启 |
