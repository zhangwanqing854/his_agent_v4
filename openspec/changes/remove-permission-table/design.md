## 上下文

**当前架构：**
```
Permission ──1:N──▶ Duty ──M:N──▶ RoleDuty ──N:1──▶ Role
   (空表)        (职责)       (关联表)        (角色)
```

Permission 表作为 Duty 的分类层，但：
- 表中无数据
- 前端按 Permission 分组显示职责 → 无分组 → 职责不显示
- 业务上 Role 直接关联 Duty 即可满足需求

**约束：**
- RoleDuty 关联表必须保留（角色-职责关联）
- 不影响现有用户权限判断逻辑
- 前后端需同步修改

## 目标 / 非目标

**目标：**
- 移除 Permission 层，简化架构为 `Duty ──M:N──▶ RoleDuty ──N:1──▶ Role`
- 职责分配界面正常显示和操作
- API 端点简化

**非目标：**
- 不新增职责分类功能
- 不修改用户认证/权限判断
- 不修改 RoleDuty 表结构

## 决策

### 决策 1：UI 显示方式

**方案对比：**
| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: 平铺显示 | 所有职责 checkbox 列表 | 简单、直观 | 职责多时可能混乱 |
| B: 保持分组 | 用硬编码分组代替数据库 | 界面不变 | 维护成本高 |
| C: 分页+分类 | 按职责类型硬编码分类 | 清晰 | 需额外设计 |

**选择：方案 A（平铺显示）**
- 职责数量有限（10 个），平铺可控
- 实现简单，无额外维护
- Checkbox 多选符合用户习惯

### 决策 2：数据库迁移策略

**选择：先删代码，再清数据库**
- 代码层面先删除 Permission 相关逻辑
- 数据库 ALTER TABLE 删除 permission_id 字段
- 无数据丢失风险（permission 表本就无数据）

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| API 端点删除影响前端 | 同步删除前端 API 调用 |
| RoleService 导入 Permission | 清理导入语句 |
| 数据库字段删除 | 先备份 schema，再执行 ALTER |
| UI 变化用户不熟悉 | 保持职责 checkbox 交互不变 |
