## 新增需求

### 需求:系统必须支持科室患者信息总览的SOAP接口调用

系统必须通过 SOAP 接口调用 HIS 系统的 `getDeptPatientInfo` 方法，获取指定科室的患者信息总览数据。接口地址必须携带有效的 access_token 参数。

#### 场景:成功获取科室患者信息总览
- **当** 系统调用 SOAP 接口并提供有效的科室编码参数（arg0: "0000094"）
- **那么** 系统必须成功接包含科室患者信息总览的 JSON 响应（包含 totalNum、newInHos、diseNum 等字段）

#### 场景:获取所有科室患者信息总览
- **当** 系统调用 SOAP 接口并提供空科室编码参数（arg0: ""）
- **那么** 系统必须成功获取所有科室的患者信息总览数据

#### 场景:SOAP接口返回错误
- **当** SOAP 接口返回错误响应或网络超时
- **那么** 系统必须记录错误日志并重试最多3次（间隔5秒），最终失败时必须抛出同步失败异常

### 需求:系统必须解析和存储科室患者信息总览数据

系统必须将 SOAP 响应中的 JSON 数据解析为结构化对象，并存储到 dept_patient_overview 数据表中。每条数据必须包含科室编码、科室ID、全科人数、新入院人数、危重人数、死亡人数、出院人数、手术人数、转入人数、转出人数和同步时间。

#### 场景:成功解析和存储科室患者信息总览
- **当** SOAP 响应包含科室患者信息总览数据（totalNum: 86, newInHos: 0, diseNum: 10, code_dept: "0000094", id_dept: "0001Z8100000000LJ1N4"）
- **那么** 系统必须创建 dept_patient_overview 记录，dept_code 为 "0000094"，dept_id 为 "0001Z8100000000LJ1N4"，total_num 为 86

#### 场景:更新已存在的科室患者信息总览
- **当** 科室编码已存在对应的 dept_patient_overview 记录
- **那么** 系统必须更新该记录的各项统计数据，而不是创建新记录

#### 场景:数据字段缺失或格式错误
- **当** SOAP 响应中数据缺少必要字段（如 totalNum 缺失）
- **那么** 系统必须跳过该数据并记录警告日志，不影响其他数据的解析和存储

### 需求:系统必须支持定时同步科室患者信息总览数据

系统必须通过定时任务自动同步科室患者信息总览数据，同步频率为每小时一次。

#### 场景:定时任务触发同步
- **当** 系统时间到达定时任务触发点（如每小时）
- **那么** 系统必须自动调用 SOAP 接口同步所有科室的患者信息总览数据

#### 场景:定时任务执行失败
- **当** 定时任务执行过程中发生异常
- **那么** 系统必须记录异常日志并保持系统稳定，不影响下一次定时任务的执行

### 需求:系统必须支持手动触发科室患者信息总览数据同步

系统必须提供 REST API `/api/sync/dept-patient-info` 支持手动触发科室患者信息总览数据同步，允许指定科室编码参数。

#### 场景:手动触发同步指定科室
- **当** 用户调用 `/api/sync/dept-patient-info` 并提供科室编码参数（deptCode）
- **那么** 系统必须立即调用 SOAP 接口同步指定科室的患者信息总览数据，并返回同步成功响应

#### 场景:手动触发同步所有科室
- **当** 用户调用 `/api/sync/dept-patient-info` 且不提供科室编码参数
- **那么** 系统必须立即调用 SOAP 接口同步所有科室的患者信息总览数据，并返回同步成功响应

#### 场景:手动触发同步失败
- **当** 手动触发同步过程中发生异常
- **那么** 系统必须返回同步失败响应（包含错误信息），并记录异常日志
