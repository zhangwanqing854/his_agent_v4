## 新增需求

### 需求:系统必须自动生成唯一交班编号

系统必须在创建交班时自动生成唯一编号，编号格式为年份后两位+月份+天+自增序号。

#### 场景:生成编号格式正确
- **当** 系统创建交班
- **那么** 生成的编号格式为 `YYMMDDSSS`（如 `260415001`）

#### 场景:编号包含日期信息
- **当** 系统在 2026年4月15日 创建交班
- **那么** 编号前6位为 `260415`

#### 场景:序号从001开始
- **当** 系统在某日创建第一个交班
- **那么** 编号序号部分为 `001`

#### 场景:序号递增
- **当** 系统在某日已创建编号 `260415001` 的交班
- **那么** 下一个交班编号为 `260415002`

#### 场景:跨日序号重置
- **当** 系统在4月15日最后一个编号为 `260415010`
- **那么** 4月16日第一个编号为 `260416001`

### 需求:系统必须保证编号全局唯一

系统必须保证交班编号全局唯一，不允许重复编号。

#### 场景:编号不可重复
- **当** 系统尝试生成编号 `260415001`
- **那么** 如果已存在该编号，系统重新生成直到获得唯一编号

#### 场景:并发创建不冲突
- **当** 多个用户同时创建交班
- **那么** 每个交班获得不同的编号

### 需求:编号必须存储到数据库

系统必须将生成的编号存储到 `shift_handover.handover_no` 字段。

#### 场景:编号持久化
- **当** 交班创建成功
- **那么** `handover_no` 字段包含生成的编号

### 需求:编号字段必须有唯一约束

数据库 `handover_no` 字段必须有 UNIQUE 约束，保证数据库层面不重复。

#### 场景:数据库唯一约束
- **当** 尝试插入重复编号
- **那么** 数据库拒绝插入并抛出唯一约束冲突错误
