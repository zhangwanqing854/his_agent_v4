## 新增需求

### 需求:系统必须提供短信发送服务

系统必须提供可扩展的短信发送服务，支持通过配置切换不同的短信平台（阿里云、腾讯云或自定义平台）。

#### 场景:配置短信平台
- **当** 系统配置 `sms.provider=aliyun`
- **那么** 系统使用阿里云短信服务发送短信

#### 场景:禁用短信功能
- **当** 系统配置 `sms.enabled=false`
- **那么** 系统不发送任何短信，所有短信发送请求被跳过

### 需求:系统必须异步发送短信

系统必须异步发送短信，短信发送失败不得阻塞或影响交班提交流程。

#### 场景:异步发送成功
- **当** 交班提交成功
- **那么** 系统异步发送短信，交班提交立即返回成功响应

#### 场景:异步发送失败
- **当** 短信发送失败（如 API 超时）
- **那么** 交班提交仍然成功，系统记录失败日志

### 需求:系统必须从 his_staff 表获取电话号码

系统发送短信时必须从 `his_staff.phone` 字段获取接班医生电话号码。

#### 场景:电话号码存在
- **当** 接班医生 `his_staff.phone` 字段有有效电话号码
- **那么** 系统向该号码发送短信

#### 场景:电话号码为空
- **当** 接班医生 `his_staff.phone` 字段为空或 null
- **那么** 系统跳过短信发送，记录日志"电话号码为空"

### 需求:系统必须记录短信发送日志

系统必须将短信发送结果记录到 `sms_log` 表，包含发送状态、时间、错误信息。

#### 场景:发送成功记录
- **当** 短信发送成功
- **那么** 系统在 `sms_log` 表记录 `status=SUCCESS`，记录 `sent_at` 时间

#### 场景:发送失败记录
- **当** 短信发送失败
- **那么** 系统在 `sms_log` 表记录 `status=FAILED`，记录 `error_message`

### 需求:短信内容必须使用固定模板格式

系统发送的短信必须使用固定模板格式，包含科室名称、交班医生姓名、患者数量。

#### 场景:短信内容格式
- **当** 系统发送短信
- **那么** 短信内容为："【北京大学国际医院】您有新的交接班任务待处理。科室：{科室名}，交班医生：{医生名}，患者数：{数量}，请及时登录系统处理。"
