## 新增需求

### 需求:系统必须支持生命体征数据的SOAP接口调用

系统必须通过 SOAP 接口调用 HIS 系统的 `getEntVt` 方法，获取指定时间范围内的生命体征数据。接口地址必须携带有效的 access_token 参数。

#### 场景:成功获取生命体征数据
- **当** 系统调用 SOAP 接口并提供有效的时间范围参数（开始时间：2026-04-01 13:04:46，结束时间：2026-04-15 13:04:46）
- **那么** 系统必须成功接包含生命体征数据的 JSON 响应（包含 code_ent、name_vs_item、value_vs_item 字段）

#### 场景:SOAP接口返回错误
- **当** SOAP 接口返回错误响应或网络超时
- **那么** 系统必须记录错误日志并重试最多3次（间隔5秒），最终失败时必须抛出同步失败异常

### 需求:系统必须解析和存储生命体征数据

系统必须将 SOAP 响应中的 JSON 数据解析为结构化对象，并存储到 vital_signs 数据表中。每条数据必须包含就诊流水号（visit_no）、体征类型、体征值、记录时间和同步时间。

#### 场景:成功解析和存储单条生命体征数据
- **当** SOAP 响应包含单条生命体征数据（code_ent: "1142214", name_vs_item: "体温", value_vs_item: "37.1"）
- **那么** 系统必须创建 vital_signs 记录，visit_no 为 "1142214"，sign_type 为 "体温"，sign_value 为 "37.1"

#### 场景:解析多条生命体征数据
- **当** SOAP 响应包含多条生命体征数据（体温、心率、血压等）
- **那么** 系统必须批量解析并存储所有数据，每条数据独立存储为 vital_signs 记录

#### 场景:数据字段缺失或格式错误
- **当** SOAP 响应中某条数据缺少必要字段（如 code_ent 缺失）
- **那么** 系统必须跳过该条数据并记录警告日志，不影响其他数据的解析和存储

### 需求:系统必须支持定时同步生命体征数据

系统必须通过定时任务自动同步生命体征数据，同步频率为每天4次（0点、6点、12点、18点）。

#### 场景:定时任务触发同步
- **当** 系统时间到达定时任务触发点（如12:00）
- **那么** 系统必须自动调用 SOAP 接口同步过去6小时的生命体征数据

#### 场景:定时任务执行失败
- **当** 定时任务执行过程中发生异常
- **那么** 系统必须记录异常日志并保持系统稳定，不影响下一次定时任务的执行

### 需求:系统必须支持手动触发生命体征数据同步

系统必须提供 REST API `/api/sync/vital-signs` 支持手动触发生命体征数据同步，允许指定时间范围参数。

#### 场景:手动触发同步成功
- **当** 用户调用 `/api/sync/vital-signs` 并提供时间范围参数（startTime、endTime）
- **那么** 系统必须立即调用 SOAP 接口同步指定时间范围内的生命体征数据，并返回同步成功响应

#### 场景:手动触发同步失败
- **当** 手动触发同步过程中发生异常
- **那么** 系统必须返回同步失败响应（包含错误信息），并记录异常日志

### 需求:系统必须存储生命体征条目的HIS唯一标识

系统必须在 vital_signs 表中存储 HIS 返回的生命体征条目唯一标识（his_id），用于精确判断重复数据。his_id 字段必须为字符类型，长度 50。

#### 场景:his_id 字段存在
- **当** vital_signs 表创建或更新
- **那么** 系统必须包含 his_id 字段，类型为 varchar(50)

#### 场景:his_id 数据流转
- **当** HIS 接口返回生命体征数据包含 his_id 字段
- **那么** 系统必须将 his.his_id 值存储到 vital_signs.his_id

## 修改需求

### 需求:系统必须采用"按his_id判断重复"同步策略

系统必须将 SOAP 响应中的 JSON 数据解析为结构化对象，并存储到 vital_signs 数据表中。同步时必须采用"按 his_id 判断重复"策略：如果 his_id 已存在，跳过该条数据；如果不存在，插入新数据。不再采用"按患者删除再插入"策略。

#### 场景:his_id已存在跳过
- **当** HIS 返回生命体征数据，其 his_id 已存在于 vital_signs 表中
- **那么** 系统必须跳过该条数据，不进行插入或更新

#### 场景:his_id不存在插入
- **当** HIS 返回生命体征数据，其 his_id 不存在于 vital_signs 表中
- **那么** 系统必须插入该条数据到 vital_signs 表

#### 场景:同一患者多条数据不互相覆盖
- **当** HIS 返回同一患者的两条生命体征数据（体温 his_id="001"，脉搏 his_id="002"）
- **那么** 系统必须分别判断每条数据的 his_id，两条数据都被正确插入，互不影响

#### 场景:his_id字段缺失
- **当** HIS 返回的生命体征数据缺少 his_id 字段
- **那么** 系统必须跳过该条数据并记录警告日志
